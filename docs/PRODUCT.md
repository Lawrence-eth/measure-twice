# Product brief — Pentimento

## One sentence

Pentimento is a 15-minute guided practice that teaches a complete
beginner how to direct and verify one AI-built project from a rough idea to a
recoverable release.

## Product promise

> **AI can make it look finished. Learn to make it trustworthy.**

Pentimento is an authored educational simulation. It is not:

- a coding course;
- a project generator or code editor;
- a catalog or ranking of AI products;
- a collection of prompt tricks;
- a real repository or deployment service; or
- evidence that one short journey establishes mastery.

The learner practices on the fictional Willow Fix Day event page. No real file,
repository, email, account, or deployment is changed.

## Audience

The primary learner has an idea but no experience with AI coding tools, project
folders, Git, GitHub, previews, hosting, APIs, or deployment. They can use a
browser and write in ordinary language.

Pentimento does not assume they know:

- which kind of AI tool to open first;
- where generated work lives;
- what a repository, commit, host, preview, or release means;
- how to verify work without reading every line of code; or
- which questions to ask before AI changes files.

The learner is still treated as capable of meaningful product, scope, evidence,
and release decisions. Plain language comes before each technical term.

## Problem

AI has removed much of the blank-page friction from software. It can also
produce a convincing interface before a beginner understands the project
underneath it.

A polished screen can hide:

- an undefined or unsupported promise;
- files that exist only in an AI conversation;
- an oversized change no one approved;
- an important visitor action no one tried;
- a deployment that serves the wrong version; and
- no known recovery path.

Product tutorials often answer with brand rankings, prompt patterns, or a
celebratory publish button. Pentimento instead teaches the durable
responsibilities that survive changing tools.

## Name and metaphor

A *pentimento* is an earlier layer visible beneath a finished work. The product
uses the metaphor literally:

> The screen is the surface. Building means directing and checking everything
> underneath it.

The persistent canvas has two learner-controlled lenses:

- **Visitor surface** — what a visitor sees and whether the important path
  works.
- **Layers underneath** — the promise, tool route, files, work agreement,
  evidence, versions, release, and recovery.

Literal explanation always precedes the artistic metaphor.

## Opening: prove the need

The welcome identifies Pentimento as a guided practice for first-time AI
builders, names its 15-minute commitment and reusable outcomes, and offers
**Start the guided project** as its dominant action. **Preview the four
chapters** is a physically separated secondary route overview.

The next screen is a one-minute worked inspection:

1. a generated Willow Fix Day preview claims it is ready to publish;
2. the learner selects **Email the organizer**;
3. they observe **Nothing happened**;
4. **Reveal what the screen hid** exposes Promise, Project home, Evidence, and
   Release through the single expressive spectral scan; and
5. **Start with layer 1 · define the promise** begins the route.

The opening also names the practice project, the eight-stop,
13-decision method, the reusable Build kit, and the
no-code/no-key/no-publishing boundary. It explains the destination without
exposing every later control.

## Learning architecture

The route groups eight stops into four chapters:

| Chapter | Stops | Learner capability |
| --- | --- | --- |
| **Shape the promise** | First version | Narrow a wishlist to one person and one complete path |
| **Ground the work** | Tools · Project home | Assign build, custody, and publishing responsibilities |
| **Direct the build** | Ask AI · Build | Bound changes and require named evidence |
| **Prove the release** | Check · Go live · Improve | Repair, release, recover, and update from a trusted source |

Across those stops, the learner makes 13 consequential decisions:

| Stop | Decisions | Saved build-kit note |
| --- | ---: | --- |
| First version | 1 | V1 boundary |
| Tools | 1 | Tool route |
| Project home | 2 | Recoverable home and runtime-AI decision |
| Ask AI | 2 | Bounded work agreement |
| Build | 1 | Evidenced saved version |
| Check | 3 | Repaired and retried visitor path |
| Go live | 2 | Checked release and recovery record |
| Improve | 1 | Source-backed update |
| **Total** | **13** | **Complete idea-to-release method** |

## Interaction model

Every stop follows:

```text
why it matters → consequential choice or trial → visible consequence →
inspect reusable build-kit note → explicitly save the layer
```

A correct answer does not auto-advance. It remains visible with:

- what changed;
- the reusable rule;
- when to use it; and
- which failure it prevents.

The learner then files one build-kit note through a meaningfully named
checkpoint. These eight checkpoints are reflection boundaries, distinct from
the 13 curriculum decisions. The eight stop-level notes accumulate into the
existing five-guide Build kit; they are not eight new guides.

Wrong choices explain a specific cost, missing proof, or additional system and
remain revisable. There are no scores, streaks, badges, mastery claims, trick
distractors, or confetti.

## Content model

Core screens keep one decision in focus but do not reduce the teaching to thin
quiz copy. Every stop states why it matters and what artifact the learner will
keep.

Optional field notes organize comprehensive content into four useful questions:

1. **Why** does this decision matter?
2. **Make** what exact artifact or action?
3. **Proof** of what observable result?
4. **Avoid** which costly failure or wasted work?

The persistent **Build kit** opens five copyable milestone guides:

1. Shape the first version.
2. Choose a route and project home.
3. Ask, inspect, and save.
4. Check and repair.
5. Release and improve.

The Build kit is available beside the current task and remains under the same
name at completion; it is not locked behind success.

## End-of-journey outcomes

After the authored route, the learner can:

1. define one safe, small, complete first version;
2. explain where AI builds, where the work survives, and where it is published;
3. choose between hosted and repository-aware routes for a stated reason;
4. separate build-time AI from runtime AI;
5. write a bounded work agreement with trusted facts and a stopping point;
6. use the ask → inspect → run → check → save loop;
7. distinguish an AI claim, changed files, preview, human path, and public path;
8. report and repair an observed defect without discarding working decisions;
9. name the released version and recovery version; and
10. update the trusted source before changing the visible page.

One completed journey is practice, not proof of lasting mastery.

## Transfer and GPT‑5.6 boundary

Completion provides route-specific first actions, the five-guide Build kit,
four durable habits, and the optional V1 brief workshop.

The V1 brief workshop first creates a complete, copyable brief locally. A
learner may then explicitly request a bounded GPT‑5.6 reflection containing one
strength, exactly two unresolved questions, one feature to postpone, one route
tradeoff, and exactly three next moves.

The model cannot grade, decide correctness, change lesson progress, generate or
execute a project, access files, request credentials, publish, or take an
external action. The deterministic authored route and local brief remain
complete without a key or model response.

## Visual direction

Pentimento uses a cool, restrained Conservation Lab language:

- lab silver and clean surface white;
- carbon and graphite for structure and instruction;
- mineral cobalt for navigation, focus, and active inspection;
- viridian for checked states and oxide for observed failure;
- ruled plates, registration marks, underlayers, and one spectral scan; and
- Instrument Sans for instruction and controls, with serif reserved for
  artifact titles and distilled rules.

Interaction and motion serve comprehension. Reduced motion preserves all
meaning. Gradients are limited to the registration grid and earned spectral
scan. The product avoids generic dashboard chrome, chat-first presentation,
glossy component gradients, hover lift, perpetual glow, confetti, badges,
decorative card shadows, and complexity that competes with instruction.

## Success signals

A first-time learner should be able to:

- explain after the worked inspection why a polished screen is not a finished
  project;
- name the promise, custody, evidence, and release layers;
- describe the build → remember → publish role map;
- explain why an AI completion message is not human-path evidence;
- reproduce the fail → bounded repair → retry method;
- identify the checked release and recovery layer; and
- use one build-kit note or Build kit template on another project.

A judge should see the complete causal arc in under three minutes. A learner
should complete the core route in about 15 minutes and explore deeper field
notes at their own pace.
