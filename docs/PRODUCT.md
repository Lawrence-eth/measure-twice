# Product brief — Measure Twice

## One sentence

Measure Twice is a consequence-driven simulation that teaches complete beginners how to direct, inspect, and verify AI while building a project.

## Audience

- Someone with an idea but no prior experience building with AI
- No coding, Git, GitHub, testing, or deployment vocabulary assumed
- Learners who need a safe first experience before touching a real project

## Problem

Most beginner AI education teaches prompts, tool tours, or passive tips. It rarely teaches the harder transferable judgment: defining an outcome, preserving trusted context, limiting a change, inspecting what happened, verifying claims, repairing from evidence, and deciding when work is actually ready.

## Product promise

The learner experiences one realistic build from polished failure to verified release. Every completed mission produces a causal evidence chain:

`decision -> AI proposal -> project consequence -> evidence -> repair -> transfer`

The central principle is:

> AI output is a proposal. Your project becomes trustworthy through evidence.

## Learning spine: TRACE

- **Target** a user-visible, testable outcome.
- **Record** trusted facts, decisions, files, and recoverable versions.
- **Assign** one bounded, reversible task with context and proof-of-done.
- **Check** the artifact independently instead of trusting the AI summary.
- **Evolve** from observed evidence, rerun checks, and save a known-good state.

## Flagship mission

**The page that looks finished**

A fictional community Repair Cafe needs a simple event page. An AI-created page looks polished and declares itself ready, but it contains an invented promise, a broken mobile action, and no evidence that checks ran. The learner must recover the project by moving through TRACE.

The experience teaches Git and GitHub as shared memory and undo history, `README.md` as durable context, safe secret handling, acceptance checks, plan-before-editing, bounded changes, diff review, mobile and keyboard checks, evidence-based bug reports, meaningful commits, deployment gates, and rollback.

## Signature interactions

- **Context X-ray** makes visible what the AI can and cannot know.
- **Blast Radius** turns extra features, dependencies, and changed files into visible risk.
- **Evidence Threads** connect each requirement to an independent check; AI claims never turn a requirement green.
- **Checkpoint Rewind** makes a reckless change recoverable.
- **Build Replay** reveals the causal value of each learner decision.

## MVP

- One deeply authored 18–20 minute mission
- Nine short scenes with meaningful consequence and repair
- One interactive artifact with authored fact and mobile defects
- Deterministic evidence engine and judge-ready demo mode
- One constrained GPT-5.6 debrief that interprets reflection but cannot alter scores
- A reusable beginner field guide and project templates
- Local progress, responsive layout, keyboard support, and reduced motion

## Non-goals for Build Week

- Authentication or learner accounts
- Real GitHub OAuth or deployment
- An arbitrary app generator or code editor
- Multiple shallow courses or a generic chatbot
- High-stakes grading or claims of mastery after one mission

## Success signal

A first-time viewer should see a polished AI artifact fail, watch the learner turn unsupported claims into verified evidence, and understand why this is deeper than prompt training in under three minutes.
