# Pentimento v5 experience contract

This is the current implementation contract for Pentimento. It supersedes
earlier screen architecture where the two conflict; dated release evidence in
[BUILD_LOG.md](BUILD_LOG.md) remains historical.

## Product in one sentence

Pentimento is a **15-minute interactive field guide** in which a complete
beginner directs one simulated AI-built project from a tempting surface to a
supportable, recoverable, checked release.

It is an authored educational simulation—not a coding course, prompt library,
tool ranking, project generator, or development environment. It never builds or
publishes a real learner project.

## Thesis

> AI can make it look finished. Learn to make it trustworthy.

AI has made a polished surface fast. It has not removed the need to decide:

- what the first version can honestly promise;
- where the work and its history survive;
- what AI may change and when it must stop;
- which observations count as evidence;
- which exact version is released; and
- how to repair or recover it.

The visual pentimento makes this relationship literal:

```text
visitor surface
    ↓ reveal
promise · custody · work agreement · evidence · release · recovery
```

## Opening: The Reveal

The first screen must answer three questions without requiring a route map:

1. **What is this?** An interactive field guide for first-time AI builders.
2. **Why does it matter?** A polished screen can hide an undefined promise,
   unrecoverable work, an untested path, and an unknown release.
3. **What will I gain?** A first-version brief, tool map, AI work agreement,
   evidence ladder, and release-and-recovery checklist.

The opening then proves the problem through interaction:

1. Show a polished generated Willow Fix Day preview with AI status **Ready to
   publish**.
2. Ask the learner to **Test the only important action**.
3. Let the learner observe that the email action does nothing.
4. Offer **Reveal the project underneath**.
5. Use a raking-light reveal to expose four missing layers:
   **Promise**, **Project home**, **Evidence**, and **Release**.
6. State the thesis: **The screen is the surface. Building means directing and
   checking everything underneath it.**

The Reveal is useful before explanation: it creates a specific contradiction
between “looks finished” and “works for a visitor.” The remaining welcome
content explains why that contradiction matters, lets the learner explore four
capabilities, identifies what Pentimento is not, and previews the four chapters.

The dominant opening action is **Test the finished-looking project**, which
moves focus to the preview instead of skipping the Reveal. After the failure,
**Learn the method that catches this** begins the route. **See exactly what you
will learn** opens the four-chapter overview, whose final action is **Start with
the first promise**. Opening safety language remains literal: **No coding · no
setup · nothing real publishes.**

## Route architecture

The route contains four chapters, eight stops, and 13 consequential decisions.
The chapters are conceptual orientation; the stops remain the saved progress
units.

| Chapter | Stops | Purpose |
| --- | --- | --- |
| **1. Shape the promise** | First version | Decide what this version can honestly finish |
| **2. Ground the work** | Tools · Project home | Give tools clear responsibilities and the work a recoverable home |
| **3. Direct the build** | Ask AI · Build | Bound AI changes and demand evidence before trust |
| **4. Prove the release** | Check · Go live · Improve | Repair what fails, release the checked layer, and update its source |

The persistent header says both **Chapter n of 4** and **Stop n of 8**. The
route dialog explains the same hierarchy without turning progress into
navigation homework.

### Stop 1 — First version

The learner chooses the smallest complete visitor path: read approved facts,
decide whether an item fits, and email a question. Booking and donation remain
plausible wrong choices because they reveal the extra operational systems they
would introduce.

Playbook note: one person, one complete path, trusted facts, and a clear **Not
now** boundary.

### Stop 2 — Tools

The learner chooses between two legitimate starting tradeoffs:

- faster preview with more dependence on one hosted service; or
- more setup with clearer files and recoverable history.

The durable map remains:

```text
AI workspace builds → project home remembers → host publishes
```

Playbook note: build responsibility, project custody, publishing responsibility,
first action, and route-specific risk.

### Stop 3 — Project home

The learner rejects a chat-only “home,” selects the route-specific recoverable
home, and decides whether the finished event page needs runtime AI. It does not:
approved facts and a normal email link create the complete visitor result.

Playbook note: route-specific files/history plus a truthful build-time-AI versus
runtime-AI boundary.

### Stop 4 — Ask AI

The learner rejects an unbounded “build and deploy everything” request, asks AI
to inspect the project and return a small plan, and approves only the first
shown step.

Playbook note: a work agreement containing context, boundaries, named evidence,
and a stopping point.

### Stop 5 — Build

The learner distinguishes AI’s completion claim and a polished preview from a
complete evidence chain. The reusable loop is:

```text
ask → inspect → run → check → save
```

Playbook note: a named saved version backed by visible files, a preview, and the
important human path.

### Stop 6 — Check

The learner activates the important contact action and observes a failure,
requests the smallest bounded repair, then repeats the same path. The repaired
version is not eligible for release until the retry succeeds.

Playbook note: observed behavior, reproduction step, expected result, preserved
behavior, repair boundary, and repeated check.

### Stop 7 — Go live

The learner chooses checked V4 over polished but untested V3, then verifies the
important path at the simulated public address. A successful deployment report
is not accepted as public-path evidence.

Playbook note: exact released version, public path, known limit, and recovery
version.

### Stop 8 — Improve

The learner updates the approved source before changing the page, repeats the
affected checks, and saves a new layer without erasing the checked release.

Playbook note: source-backed change, affected path, evidence repeated, and prior
known-good layer.

## Interaction grammar: decide → inspect → save

Every stop follows one causal learning loop:

1. **Orient** — name why the decision matters and what artifact it will create.
2. **Predict, choose, or try** — require one consequential learner action.
3. **Observe** — keep the causal consequence and canvas change visible.
4. **Inspect the Playbook note** — show the reusable rule, when to use it, and the
   failure it prevents.
5. **Save the layer** — require an explicit, meaningful checkpoint.
6. **Continue** — reveal the next stop only after the checkpoint.

The 13 decisions are the curriculum interactions. The eight checkpoint buttons
are deliberate reflection boundaries, not quiz questions: they give the
learner time to read and save one stop-level Playbook note that would otherwise
disappear during auto-advance. Those eight notes accumulate into the existing
five-card take-home Playbook; they are not eight additional cards.

Checkpoint labels must name the artifact:

- **Keep this V1 boundary**
- **Save this tool route**
- **Pin this project home**
- **Approve this work agreement**
- **Save this evidenced version**
- **Pin the repaired path**
- **Record this checked release**
- **Save this source-backed update**

Wrong choices remain recoverable and explain the specific cost, missing proof,
or extra system they would introduce. Feedback does not use scores, streaks,
badges, mastery claims, or trick distractors.

## Content hierarchy

Pentimento must be detailed without forcing every detail into the required
path.

### Always visible

- why the current decision matters;
- what the learner will leave with;
- one question or trial;
- its causal consequence;
- the current project surface or underlayers;
- the Playbook-note checkpoint; and
- access to the Field Guide.

### Available in context

Every stop has a field note organized as:

- **Why** — the decision this knowledge changes;
- **Make** — the concrete action or template;
- **Proof** — the observable output;
- **Avoid** — the wasted work or failure it prevents.

Route-specific instructions, examples, prompt anatomy, evidence limits, release
states, and update rules belong here. Optional depth must remain useful on its
own and must never be required to guess a core answer.

### Persistent Field Guide

The **Field guide** button is available throughout the journey. It opens five
copyable milestone cards:

1. Shape the first version.
2. Choose a route and project home.
3. Ask, inspect, and save.
4. Check and repair.
5. Release and improve.

Each card states when to use it, what to do, what counts as proof, and what
template to copy. It is the same reference called the **5-card Playbook** at
completion; it is not a reward locked behind the route.

## Surface and underlayers canvas

The persistent canvas has two explicit lenses:

- **Visitor surface** shows what a visitor currently sees and whether the
  important action works.
- **Project underneath** shows the saved brief, route, files, work agreement,
  evidence, versions, release proof, and recovery.

Every successful stop changes this canvas. Completed layers remain visible as
earlier paint rather than disappearing behind the latest state. Both lenses
have complete text equivalents and button controls; no meaning depends on
dragging, color, animation, or hover.

On mobile the canvas remains open and reachable. The central Pentimento
metaphor must not be hidden by default merely because the screen is narrow.

## Evidence model

Pentimento separates five levels:

```text
AI claim → changed files → preview → human path → public path
```

Each proves something different:

- an AI claim reports what the model believes happened;
- changed files show that project material moved;
- a preview shows a rendered candidate;
- a human path shows the important task working in the checked environment;
- a public path shows the selected version working at the visitor address.

No lower level may be described as proof of a higher one. A deployment success
does not prove the public path, and a passing technical check does not prove the
visitor can finish.

## Completion and transfer

Completion leads with the learner's new capability: guide a project from idea
to evidence. It provides:

- route-specific first actions for starting today;
- the persistent five-card Playbook;
- four durable habits;
- the completed surface/underlayers canvas; and
- the optional Teaching Mirror for shaping the learner's own V1 brief.

The Teaching Mirror remains optional, bounded, and separate from curriculum
truth. The deterministic route works without a model, key, account, or network
request. GPT‑5.6 may reflect on a deliberately submitted brief; it cannot grade,
change progression, access files, build, publish, or perform an external
action.

## Accessibility and responsive contract

- Semantic landmarks, headings, lists, forms, dialogs, buttons, and status
  messages match their meaning.
- Every control has an accessible name, visible focus, and keyboard operation.
- Correct feedback and checkpoint focus are available long enough to inspect.
- Dialogs contain focus, support safe dismissal, and restore focus.
- Selection, failure, saved state, and release state never depend on color
  alone.
- Controls used at narrow widths provide practical touch targets.
- The route works at 320px, 390px, 768px, desktop, and 200% zoom without
  task-blocking horizontal scrolling.
- Reduced motion removes decorative transitions while preserving every state
  change.
- The Reveal, surface/underlayers canvas, evidence ladder, and release sequence
  all have text equivalents.

## Functional and safety boundaries

Pentimento does not:

- create, edit, execute, or publish a real learner project;
- connect to or mutate GitHub;
- send the simulated email;
- create accounts, databases, payments, or API integrations;
- request learner credentials;
- require runtime AI for the Willow Fix Day page; or
- claim that one completed simulation proves mastery.

All Willow Fix Day files, people, addresses, versions, public URLs, releases,
and external actions are fictional or simulated.

## Acceptance evidence

Do not infer new verification results from an earlier release. Before v5 is
described as deployed or verified, its exact commit and artifact must pass the
repository's proportionate checks and the result must be added to
[BUILD_LOG.md](BUILD_LOG.md). Historical v1–v4 evidence remains evidence only
for those exact commits.
